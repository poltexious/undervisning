# F2.10 – Unity-MVP: prototype med sortering som mekanik

!!! info "Om forløbet"

    - **Timer:** 12 (F-semester)
    - **Studie+-forløb:** SP8 — Ship It: Unity-MVP med sortering som mekanik
    - **Bog:** *The C# Player's Guide*, Level 32 (anvendt)

!!! abstract "Læringsmål"

    Efter forløbet kan du:

    - bruge programmering til at undersøge og behandle et selvvalgt problemområde i Unity
    - anvende avancerede konstruktioner og arbejde inkrementelt med arkitekturen i et Unity-projekt
    - designe testcases, der dækker normale forløb og edge cases
    - commit'e dit arbejde regelmæssigt og pushe det til GitHub

## Om forløbet

*Skriv motivation: dette er årets store Unity-projektforløb — alle Unity-emner fra F2.4-F2.5 samles, og sorteringsalgoritmerne fra F2.6-F2.9 genbruges som en ÆGTE gameplay-mekanik (ikke bare en isoleret øvelse). Git introduceres som procesdokumentation sideløbende med devlogs.*

## Lektionsplan

*Skriv time-for-time-plan for forløbets 12 timer: idé/MVP-afgrænsning → iterativ opbygning → integrering af sortering som mekanik → test → Git-historik gennem hele forløbet.*

## Teori

*Skriv teoriafsnit (kort, da det meste er anvendelse af tidligere lært stof) om:*

- *hvordan sortering kan blive en SYNLIG gameplay-mekanik, ikke kun en baggrundsberegning*
- *manuelle testcases og edge cases (4.3)*
- *Git: `init`, `add`, `commit`, `push`, `pull`, GitHub som remote, branches (intro), commit-beskeder som dokumentation (4.4)*

## Øvelser

MVP-prototype hvor sortering ER en gameplay-mekanik.

### :green_circle: Højscore-sortering

*En highscore-tavle, der sorteres og vises efter score.*

### :yellow_circle: Magnet-powerup

*En magnet-powerup, der finder og samler den nærmeste coin ved hjælp af sortering/søgning.*

### :red_circle: Tower-targeting eller egen idé

*Et tower-defense-tårn, der målretter den hurtigste/nærmeste fjende — eller en selvvalgt idé af tilsvarende sværhedsgrad.*

## Leverance

MVP-prototype med sortering integreret som gameplay-mekanik, Git-historik der viser processen, og devlogs gennem hele forløbet.
